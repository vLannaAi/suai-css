## Stage 1 → Stage 2: Adding Utilities

**Goal**: Add UnoCSS utilities while keeping existing HTML semantic.

### Step 1: Install Dependencies

```bash
npm install -D unocss @suai/css
# or
pnpm add -D unocss @suai/css
```

### Step 2: Configure Build Tool

**Vite** (recommended):

```js
// vite.config.js
import { defineConfig } from 'vite'
import UnoCSS from 'unocss/vite'

export default defineConfig({
  plugins: [
    UnoCSS()
  ]
})
```

**Webpack**:

```js
// webpack.config.js
const UnoCSS = require('@unocss/webpack').default

module.exports = {
  plugins: [
    UnoCSS()
  ]
}
```

### Step 3: Create UnoCSS Config

```js
// uno.config.ts
import { defineConfig, presetUno } from 'unocss'
import { presetSUAI } from '@suai/css'

export default defineConfig({
  presets: [
    presetUno(),
    presetSUAI()
  ]
})
```

### Step 4: Update HTML Imports

**Before (Stage 1)**:
```html
<link rel="stylesheet" href="suai-html/suai.css">
```

**After (Stage 2)**:
```html
<link rel="stylesheet" href="suai-css/suai.css">
<script src="uno.css"></script> <!-- Generated by UnoCSS -->
```

### Step 5: Enhance Components Gradually

**Stage 1 HTML** (still works):
```html
<article>
  <h2>Article Title</h2>
  <p>Content here</p>
  <button>Read More</button>
</article>
```

**Stage 2 Enhanced** (add utilities where needed):
```html
<article class="max-w-prose mx-auto">
  <h2 class="text-3xl font-bold mb-4">Article Title</h2>
  <p class="text-gray-600 leading-relaxed">Content here</p>
  <button class="px-6 py-3 rounded-lg hover:shadow-xl transition">
    Read More
  </button>
</article>
```

### Step 6: Use suai-type Attributes

For component-level styling:

```html
<div suai-type="card" class="p-6 shadow-lg">
  <h3 suai-type="title1">One-line Title</h3>
  <p>This is a card with semantic type attribute.</p>
</div>
```

### Migration Tips

✅ **Do**:
- Migrate one page at a time
- Keep Stage 1 syntax for simple content
- Add utilities only where customization needed
- Use `suai-type` for reusable patterns

❌ **Don't**:
- Replace all HTML at once
- Add utilities everywhere (only where needed)
- Remove semantic HTML structure
- Forget about Stage 1 fallbacks

### Rollback Strategy

If you need to revert:

1. Remove UnoCSS from package.json
2. Remove UnoCSS config
3. Switch back to Stage 1 CSS link
4. Remove utility classes from HTML (semantic HTML still works)

## Stage 2 → Stage 3: Adding Interactivity

**Goal**: Add Web Components for interactive features while maintaining fallbacks.

### Step 1: Install Stage 3

```bash
npm install @suai/js
# or
pnpm add @suai/js
```

### Step 2: Import Web Components

```js
// main.js
import '@suai/js'

// Or import specific components
import { SuaiTabs, SuaiSelect, SuaiDialog } from '@suai/js'
```

### Step 3: Wrap with Progressive Enhancement

**Before (Stage 2)**:
```html
<div suai-type="tabs" class="...">
  <div role="tablist">
    <button role="tab">Tab 1</button>
    <button role="tab">Tab 2</button>
  </div>
  <div role="tabpanel">Content 1</div>
  <div role="tabpanel">Content 2</div>
</div>
```

**After (Stage 3)** - with fallback:
```html
<suai-tabs>
  <!-- Stage 2 fallback still works without JS -->
  <div suai-type="tabs" class="...">
    <div role="tablist">
      <button role="tab">Tab 1</button>
      <button role="tab">Tab 2</button>
    </div>
    <div role="tabpanel">Content 1</div>
    <div role="tabpanel">Content 2</div>
  </div>
</suai-tabs>
```

### Step 4: Enhance Forms

**Before (Stage 2)**:
```html
<select name="country" class="...">
  <option value="th">Thailand</option>
  <option value="us">United States</option>
</select>
```

**After (Stage 3)**:
```html
<suai-select searchable>
  <!-- Fallback to native select -->
  <select name="country" class="...">
    <option value="th">Thailand</option>
    <option value="us">United States</option>
  </select>
</suai-select>
```

### Step 5: Add Dialogs/Modals

**Stage 3 with native fallback**:
```html
<suai-dialog id="confirm-dialog">
  <dialog>
    <h2>Confirm Action</h2>
    <p>Are you sure?</p>
    <form method="dialog">
      <button value="cancel">Cancel</button>
      <button value="confirm">Confirm</button>
    </form>
  </dialog>
</suai-dialog>

<button onclick="document.getElementById('confirm-dialog').showModal()">
  Open Dialog
</button>
```

### Migration Tips

✅ **Do**:
- Always provide Stage 2 fallback inside Web Components
- Test with JavaScript disabled
- Use progressive enhancement pattern
- Enhance incrementally (one component type at a time)

❌ **Don't**:
- Remove fallback HTML
- Assume JavaScript is always available
- Replace all components at once
- Break Stage 2 functionality

### Testing Progressive Enhancement

```bash
# Test with JS disabled in browser DevTools
# Or use Playwright
npm install -D @playwright/test

# test/progressive-enhancement.spec.ts
test('components work without JavaScript', async ({ page, context }) => {
  await context.setJavaScriptEnabled(false)
  await page.goto('/your-page')
  // Test that Stage 2 fallback works
})
```

### Rollback Strategy

1. Remove `@suai/js` from imports
2. Web Components become inert `<suai-*>` tags
3. Fallback HTML (Stage 2) continues working
4. No breakage because of progressive enhancement

## Stage 3 → Stage 4: Adopting Vue

**Goal**: Convert to Vue 3 components for maximum developer experience.

**Note**: This is a significant migration involving framework adoption.

### Step 1: Setup Vue Project

```bash
# New Vue project
npm create vue@latest

# Or add to existing
npm install vue @suai/vue
```

### Step 2: Configure SUAI Vue

```js
// main.ts
import { createApp } from 'vue'
import SUAI from '@suai/vue'
import '@suai/vue/style.css'

const app = createApp(App)
app.use(SUAI, {
  theme: 'pla',
  mode: 'light'
})
app.mount('#app')
```

### Step 3: Convert Components

**Before (Stage 3 - Web Components)**:
```html
<suai-select searchable multi>
  <select name="tags" multiple>
    <option value="1">Tag 1</option>
    <option value="2">Tag 2</option>
  </select>
</suai-select>
```

**After (Stage 4 - Vue Components)**:
```vue
<template>
  <SUSelect
    v-model="selectedTags"
    type="multi"
    :options="tagOptions"
    searchable
  />
</template>

<script setup lang="ts">
import { ref } from 'vue'
import { SUSelect } from '@suai/vue'

const selectedTags = ref<string[]>([])
const tagOptions = [
  { value: '1', label: 'Tag 1' },
  { value: '2', label: 'Tag 2' }
]
</script>
```

### Step 4: Replace Tabs

**Before (Stage 3)**:
```html
<suai-tabs>
  <div role="tablist">
    <button role="tab">Tab 1</button>
  </div>
  <div role="tabpanel">Content</div>
</suai-tabs>
```

**After (Stage 4)**:
```vue
<template>
  <SUTabs v-model="activeTab">
    <SUTab value="tab1" label="Tab 1">
      <p>Content</p>
    </SUTab>
    <SUTab value="tab2" label="Tab 2">
      <p>More content</p>
    </SUTab>
  </SUTabs>
</template>

<script setup>
import { ref } from 'vue'
import { SUTabs, SUTab } from '@suai/vue'

const activeTab = ref('tab1')
</script>
```

### Step 5: Leverage Vue Features

**Composition API**:
```vue
<script setup lang="ts">
import { ref, computed } from 'vue'
import { SUCard, SUButton } from '@suai/vue'

const count = ref(0)
const doubled = computed(() => count.value * 2)

function increment() {
  count.value++
}
</script>

<template>
  <SUCard>
    <p>Count: {{ count }}</p>
    <p>Doubled: {{ doubled }}</p>
    <SUButton @click="increment">Increment</SUButton>
  </SUCard>
</template>
```

### Migration Strategy

#### Option A: Big Bang (Rewrite)

Suitable for:
- Small to medium projects
- Starting fresh
- Team experienced with Vue

Steps:
1. Create new Vue 3 project
2. Copy assets/styles
3. Rewrite components in Vue
4. Migrate page by page

#### Option B: Gradual (Hybrid)

Suitable for:
- Large projects
- Can't stop development
- Mixed framework experience

Steps:
1. Keep Stage 3 for most pages
2. Create new features in Stage 4 (Vue)
3. Run both side-by-side
4. Migrate critical paths first
5. Eventually consolidate to Stage 4

**Hybrid setup example**:
```js
// main.js - Mount Vue only in #app
import { createApp } from 'vue'
import { SUAI } from '@suai/vue'

// Only pages with #app div get Vue
const app Element = document.getElementById('app')
if (appElement) {
  const app = createApp(App)
  app.use(SUAI)
  app.mount('#app')
}

// Stage 3 Web Components still work elsewhere
```

### Migration Tips

✅ **Do**:
- Plan the migration (don't rush)
- Train team on Vue 3 first
- Use TypeScript for type safety
- Migrate feature by feature
- Keep Stage 3 as fallback during transition

❌ **Don't**:
- Migrate everything at once (too risky)
- Skip training/documentation
- Remove Stage 3 until fully migrated
- Ignore TypeScript benefits

### Rollback Strategy

If Vue migration fails:
1. Remove `@suai/vue` dependency
2. Revert to Stage 3 Web Components
3. Stage 3 markup still works (no data loss)
4. Incrementally fix converted components

## Downgrading (Simplifying)

Sometimes you need to simplify. Here's how to go back.

### Stage 4 → Stage 3: Removing Vue Dependency

**Why?**:
- Framework-agnostic requirement emerged
- Vue bundle too large
- Team prefers vanilla JS

**Steps**:
1. Remove Vue from package.json
2. Convert Vue components to Web Components
3. Replace `<SU*>` with `<suai-*>`
4. Add fallback HTML inside Web Components

### Stage 3 → Stage 2: Removing JavaScript

**Why?**:
- Don't need interactivity after all
- JavaScript causing issues
- Want simpler stack

**Steps**:
1. Remove `@suai/js` import
2. Remove `<suai-*>` wrappers
3. Keep fallback HTML (Stage 2)
4. Styles still work

### Stage 2 → Stage 1: Removing Build Tools

**Why?**:
- Static site generator handles utilities differently
- Don't need customization
- Want zero-build simplicity

**Steps**:
1. Remove UnoCSS from package.json
2. Remove utility classes from HTML
3. Link to Stage 1 CSS directly
4. Semantic HTML still styled correctly

**Result**: You're back to pure HTML + CSS with no build process.

## Partial Migrations

You don't have to migrate an entire project. Mix stages!

### Example 1: Content + App Hybrid

```
/                 → Stage 1 (marketing homepage)
/blog/*           → Stage 1 (blog content)
/docs/*           → Stage 1 (documentation)
/app/*            → Stage 4 (Vue SPA dashboard)
/interactive/*    → Stage 3 (Web Components)
```

**Implementation**:
```html
<!-- / and /blog/* -->
<link rel="stylesheet" href="/suai-html/suai.css">

<!-- /app/* -->
<div id="app"></div>
<script src="/app-bundle.js"></script> <!-- Vue Stage 4 -->

<!-- /interactive/* -->
<script src="/suai-js/components.js"></script> <!-- Stage 3 -->
```

### Example 2: Progressive Page Enhancement

Single page with multiple stages:

```html
<!DOCTYPE html>
<html data-theme="pla" data-mode="light">
<head>
  <!-- Stage 1 base -->
  <link rel="stylesheet" href="/suai-html/suai.css">

  <!-- Stage 2 utilities (optional, per-page) -->
  <link rel="stylesheet" href="/uno.css">

  <!-- Stage 3 Web Components (optional, per-page) -->
  <script src="/suai-js/components.js"></script>
</head>
<body>
  <!-- Stage 1: Header -->
  <header>
    <h1>Site Title</h1>
    <nav>...</nav>
  </header>

  <!-- Stage 2: Hero with utilities -->
  <section class="hero bg-gradient-to-r from-blue-500 to-purple-600">
    <h2 class="text-5xl font-bold">Welcome</h2>
  </section>

  <!-- Stage 3: Interactive form -->
  <section>
    <suai-select searchable>
      <select>...</select>
    </suai-select>
  </section>

  <!-- Stage 1: Footer -->
  <footer>
    <p>&copy; 2025</p>
  </footer>
</body>
</html>
```

### Example 3: Component-Level Staging

Choose stage per component:

```html
<!-- Simple card: Stage 1 -->
<article>
  <h3>Basic Card</h3>
  <p>Content</p>
</article>

<!-- Styled card: Stage 2 -->
<article class="p-6 shadow-xl rounded-lg">
  <h3 class="text-2xl font-bold">Styled Card</h3>
  <p>More content</p>
</article>

<!-- Interactive card: Stage 3 -->
<suai-accordion>
  <article>
    <h3>Collapsible Card</h3>
    <div>Hidden content</div>
  </article>
</suai-accordion>
```

## Migration Checklist

### Before Migrating

- [ ] Identify which pages/components need enhancement
- [ ] Review team's skill level with target stage
- [ ] Assess build tool requirements
- [ ] Plan rollback strategy
- [ ] Create test cases for critical paths
- [ ] Document custom patterns

### During Migration

- [ ] Migrate incrementally (one page/component at a time)
- [ ] Test each step
- [ ] Maintain fallbacks (progressive enhancement)
- [ ] Update documentation
- [ ] Train team on new patterns
- [ ] Monitor performance impact

### After Migration

- [ ] Remove old dependencies (if fully migrated)
- [ ] Update documentation
- [ ] Clean up unused code
- [ ] Run performance audit
- [ ] Gather team feedback
- [ ] Document lessons learned

## Common Pitfalls

### ❌ Pitfall 1: Migrating Everything at Once

**Problem**: High risk, hard to rollback, team overwhelmed.

**Solution**: Migrate incrementally. One page or component at a time.

### ❌ Pitfall 2: Removing Fallbacks

**Problem**: Breaking progressive enhancement, losing accessibility.

**Solution**: Always keep Stage 1/2 fallbacks inside Stage 3 components.

### ❌ Pitfall 3: Ignoring Build Complexity

**Problem**: Build times increase, CI/CD breaks, team struggles.

**Solution**: Plan build pipeline changes. Test build process early.

### ❌ Pitfall 4: Over-Engineering

**Problem**: Using Stage 4 for a simple blog.

**Solution**: Choose the simplest stage that meets needs. Don't over-engineer.

### ❌ Pitfall 5: No Testing Strategy

**Problem**: Regressions, broken fallbacks, accessibility issues.

**Solution**: Write tests before migrating. Test with JS disabled.
